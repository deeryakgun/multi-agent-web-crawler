# Production Deployment Recommendation

## Agent Crawler v2 — Scaling Beyond a Single Machine

The current system is intentionally designed for **single-machine operation** with minimal external dependencies (SQLite + Python stdlib). Deploying it to production at scale would require changes at three layers: storage, compute, and observability.

**Storage and concurrency.** SQLite's WAL mode handles concurrent reads well, but its single-writer model becomes a bottleneck when many crawlers run simultaneously. In production, the `word_index` and `pages` tables should migrate to **PostgreSQL** (with `pg_trgm` for fast text search) or a dedicated search engine such as **Elasticsearch** / **OpenSearch**. The crawler's BFS frontier, currently an in-process `queue.Queue`, should become an external **Redis List** or **Apache Kafka** topic so that multiple crawler workers running on separate processes or pods can share a single, coordinated URL frontier. This change also unlocks horizontal scaling: you can run `N` crawler pods, all consuming from the same Kafka partition, without duplicating crawl work — provided URL deduplication uses a shared **Redis Bloom Filter** instead of a per-process in-memory set.

**Rate limiting and orchestration.** The Token Bucket should move to a distributed implementation (e.g., the Redis `INCR` + `EXPIRE` pattern or the `redis-cell` module) so that all crawler workers honour a per-domain request rate collectively. Crawler jobs themselves should be managed by a workflow orchestrator such as **Celery + Beat** or **Apache Airflow**, which would add job persistence, retry logic, and scheduled re-crawls. For the search API, adding a lightweight caching layer (**Redis** with a short TTL on hot queries) reduces SQLite/Postgres load significantly during peak traffic. Finally, the Flask development server should be replaced with **Gunicorn** (multi-worker WSGI) behind **Nginx** as a reverse proxy, with structured **JSON logging** shipped to a log aggregation stack (ELK or Loki) for real-time operational monitoring and alerting.
